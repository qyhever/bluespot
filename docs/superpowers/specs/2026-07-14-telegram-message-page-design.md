# Telegram 消息发送页面设计

## 目标

在 PC Web 端新增一个发送 Telegram 消息的表单页面。页面沿用现有邮件发送页的布局和交互模式，通过前端统一请求层调用后端 `POST /api/telegram` 接口。

## 范围

- 新增 Telegram 消息发送页面及页面私有请求服务。
- 在 Vue Router 中注册页面路由。
- 在侧边菜单的“通知”分组中提供页面入口。
- 不修改后端接口，不抽取邮件与 Telegram 的公共表单组件，也不增加富文本、附件、收件人或发送历史功能。

## 路由与菜单

- 页面路由：`/notification/tg`。
- 路由名称：`telegram-send`。
- 页面组件：`src/views/telegram/SendTelegram.vue`。
- 菜单分组沿用“通知”，子项显示为“发送 Telegram 消息”，指向 `/notification/tg`。

## 页面设计

页面视觉结构与 `src/views/mail/SendMail.vue` 保持一致：浅灰页面背景、白色表单面板、顶部标题与说明、顶部对齐的表单标签，以及右对齐的操作按钮。

表单仅包含一个“消息内容”多行文本域。文本域为必填项，并提供清晰的输入提示。操作区包含“重置”和“发送消息”两个按钮；移动端延续邮件页的纵向全宽按钮布局。

页面初始消息内容为空。点击重置时清空内容。发送过程中禁用文本域与重置按钮，发送按钮展示加载状态，以避免重复提交。

## 请求与数据流

页面同级 `service.ts` 定义请求类型和 `sendTelegramMessage` 函数，并复用 `src/utils/request.ts` 导出的 `post`：

```ts
post<void>('/telegram', { text })
```

统一请求层的 baseURL 为 `/bluespot/api`，开发代理会将请求转发到后端 `/api/telegram`。页面不会硬编码后端地址，也不会直接调用底层 `fetch`。

提交前由 TDesign 表单规则校验内容必填。请求参数中的 `text` 在发送前去除首尾空白；仅包含空白字符的内容不能提交。

## 结果反馈与错误处理

- 发送成功：显示“Telegram 消息发送成功”，随后清空消息内容。
- 校验失败：由表单在消息内容字段下展示必填提示，不发起请求。
- 请求失败：页面恢复可编辑状态；具体错误提示继续由统一请求封装处理，页面不重复弹出错误消息。

## 可访问性与响应式

- 使用 TDesign 表单标签将字段名称与输入控件建立明确关联。
- 禁用态和加载态同时表达提交状态，不仅依赖颜色。
- 保留可见的键盘焦点样式和原生表单提交行为。
- 在窄屏下缩小页面边距，操作按钮纵向排列并占满可用宽度，避免横向滚动。

## 验证

- 运行前端类型检查与生产构建，确认路由懒加载、组件类型和请求类型正确。
- 运行项目 lint，确认新增代码符合现有规范。
- 检查以下交互：空内容不可发送；有效内容只提交一次；提交期间控件禁用；成功后提示并清空；失败后恢复表单；桌面端与移动端布局正常。
